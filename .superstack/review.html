A Medium Security / DoS programs/halcyon_kernel/src/instructions/oracle/write_regression.rs:62-65 Window timestamps have no upper bound. The handler only checks end > start. A regression keeper (or one whose key is compromised) can write window_end_ts = i64::MAX on v0 init or on any write. After that, strict monotonicity (line 73–76) rejects every subsequent write — the account is permanently DoS'd. 
// before the v0/v1 branch
require!(
    args.window_start_ts > 0
        && args.window_end_ts <= now.saturating_add(REGRESSION_WINDOW_GRACE_SECS),
    HalcyonError::OracleTimestampOutOfRange
);
Pick REGRESSION_WINDOW_GRACE_SECS to match clock-skew tolerance (a few minutes).
B Medium Compute programs/halcyon_sol_autocall/src/pricing.rs:36-46, 139, 158, 178, 185 cu_trace fires on mainnet. The #[cfg(target_os = "solana")] inside cu_trace ensures it's a no-op only on host builds — BPF mainnet is still target_os = "solana", so every issuance pays for 5 sol_log_ + sol_log_compute_units_ calls. The cu_diagnostics feature exists in Cargo.toml but isn't used. 
#[inline(always)]
pub(crate) fn cu_trace(label: &str) {
    #[cfg(all(target_os = "solana", feature = "cu_diagnostics"))]
    unsafe {
        sol_log_(label.as_ptr(), label.len() as u64);
        sol_log_compute_units_();
    }
    #[cfg(not(all(target_os = "solana", feature = "cu_diagnostics")))]
    {
        let _ = label;
    }
}
Do not add cu_diagnostics to default. Mainnet build stays quiet; local profiling runs with --features cu_diagnostics.
